Foundations · 00

🧮What state management is

"State" is just data that changes over time and affects what the user sees. State management is the discipline of deciding where that data lives, who can change it, and how components stay in sync when it changes.

What counts as state?

Anything your UI remembers: the value typed in an input, whether a modal is open, the logged-in user, the items in a cart, the list of products from an API, the current theme. When state changes, React re-renders the components that depend on it — that's the entire engine behind a dynamic UI.

The four kinds of state (the most important mental model)

Picking the right tool starts with knowing what kind of state you have. This single distinction prevents most state-management mistakes:

KindExamplesWhere it usually belongs
Local UI stateinput text, modal open, toggle, hoveruseState in the component
Shared/global client statetheme, auth user, language, sidebarContext / Zustand / Redux
Server stateproducts, posts, anything from an APITanStack Query / RTK Query
URL statefilters, page number, search termThe router / query params
🧠
The biggest insight in modern state management: server data is NOT the same as client state. It's a cached copy of remote data with its own concerns (loading, staleness, refetching). When you move server data out of a global store into a dedicated tool (TanStack/RTK Query), your "global state" often shrinks to almost nothing — sometimes you don't need Redux at all. We'll come back to this repeatedly.

The problem all these libraries solve

React passes data down through props. That's fine until two distant components need the same data — say a header showing the cart count and a cart page. You'd have to thread that data through every component in between (prop drilling, Section 02). State management libraries give you a shared place any component can read from and write to directly, skipping the threading.

The shared whiteboard

Props are like passing a note hand-to-hand down a row of people — tedious and everyone in between has to handle it. A global store is a whiteboard on the wall: anyone who needs the info just looks at it, and when someone updates it, everyone watching sees the change instantly. State libraries are different designs of that whiteboard.

What we'll cover

This course walks from the built-in tools (useState, useReducer, Context) up through the Redux family (Redux, Redux Toolkit, Saga, RTK Query) and the modern minimalist libraries (Zustand, Jotai, and others) — ending with how to choose. Every tool answers the same question differently: where does shared state live and how do components subscribe to it?

Foundations · 01

⚙️useState & useReducer

Before any library, React has two built-in hooks for state. Master these first — most libraries are just these ideas scaled up, and a huge amount of state never needs anything more.

useState - the workhorse

useState stores a single piece of local state and gives you a setter to change it. Calling the setter re-renders the component with the new value.

React
import { useState } from 'react'

function Counter() {
  // [current value, function to update it]
  const [count, setCount] = useState(0)   // 0 = initial value

  return (
    <button onClick={() => setCount(count + 1)}>
      Count: {count}
    </button>
  )
}
⚠️
Update with a function when the new value depends on the old one: setCount(c => c + 1) instead of setCount(count + 1). State updates are batched and asynchronous, so reading count directly can be stale. The function form always gets the latest value.

useReducer - for complex state logic

When state has multiple sub-values that change together, or the update logic gets complicated, useReducer centralizes it. You dispatch actions (objects describing what happened), and a reducer function decides the next state. This is the exact mental model Redux is built on — learn it here and Redux is half-learned.

React
import { useReducer } from 'react'

// reducer: (currentState, action) => newState. A PURE function.
function reducer(state, action) {
  switch (action.type) {
    case 'increment': return { count: state.count + 1 }
    case 'decrement': return { count: state.count - 1 }
    case 'reset':     return { count: 0 }
    default:          return state
  }
}

function Counter() {
  const [state, dispatch] = useReducer(reducer, { count: 0 })

  return (
    <div>
      Count: {state.count}
      <button onClick={() => dispatch({ type: 'increment' })}>+</button>
      <button onClick={() => dispatch({ type: 'reset' })}>reset</button>
    </div>
  )
}

Three concepts to remember (they reappear everywhere)

  • Action — a plain object describing what happened: { type: 'increment' }. Sometimes carries data in a payload.
  • Dispatch — the function you call to send an action.
  • Reducer — a pure function (state, action) => newState. Given the same inputs it always returns the same output, and it never mutates the old state — it returns a new one.
✅
useState vs useReducer: use useState for simple, independent values. Reach for useReducer when several values change together, when the next state depends on complex rules, or when you want all update logic in one testable place. Both are local — they don't share across distant components, which is what the rest of this course solves.

Foundations · 02

🪓The prop-drilling problem

To understand why every state library exists, you need to feel the pain they remove. That pain is prop drilling.

What prop drilling is

When a deeply nested component needs data that lives high up, you must pass it through every component in between — even ones that don't use it, just to forward it along. Those middle components become cluttered "mail carriers" for data they don't care about.

React - the user prop drilled 3 levels deep
function App() {
  const [user] = useState({ name: 'Alice' })
  return <Layout user={user} />          // Layout doesn't use user...
}
function Layout({ user }) {
  return <Header user={user} />          // ...just forwards it...
}
function Header({ user }) {
  return <Avatar user={user} />          // ...still forwarding...
}
function Avatar({ user }) {
  return <span>{user.name}</span>          // finally used here!
}
🚨
Why it hurts at scale: add a field and you edit every layer. Rename a prop and you touch them all. Middle components can't be reused elsewhere because they demand props they don't need. Multiply across dozens of shared values and the app becomes rigid and noisy. This is the exact problem Context and every store library solve.

The two families of solutions

  • Context API (built into React) — create a shared "channel" any descendant can read from, no threading. Great for low-frequency global values. (Section 03.)
  • Store libraries (Redux, Zustand, Jotai…) — an external store any component subscribes to, with extra powers: fine-grained updates, devtools, middleware, async handling. (The rest of the course.)
💡
Don't over-correct. Prop drilling 1–2 levels is completely fine — reaching for a global store to avoid passing one prop one level is over-engineering. The problem is deep drilling of widely-shared data. Also, component composition (passing JSX as children) can eliminate a lot of drilling without any library at all.
EX 02Feel the pain
  • Build a 3-level component tree that drills one prop, then note the friction of adding a second.
  • Identify a case where 1-level drilling is fine and a store would be overkill.
  • Sketch how component composition (children) could avoid some drilling.

Context · 03

📡Context API

Context is React's built-in answer to prop drilling. It lets you create a shared value that any descendant component can read directly, without passing props through every level. No library to install.

The three pieces

  • createContext — creates the context object (the "channel").
  • Provider — wraps part of your tree and supplies the value to everything inside.
  • useContext — the hook a component calls to read the value.
React - a theme context
import { createContext, useContext, useState } from 'react'

// 1. create the context
const ThemeContext = createContext()

// 2. a provider component that holds the state & supplies it
export function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light')
  const toggle = () => setTheme(t => t === 'light' ? 'dark' : 'light')

  return (
    <ThemeContext.Provider value={{ theme, toggle }}>
      {children}
    </ThemeContext.Provider>
  )
}

// 3. a tiny custom hook so components consume it cleanly
export const useTheme = () => useContext(ThemeContext)
React - using it
// wrap the app once (usually near the root)
<ThemeProvider>
  <App />
</ThemeProvider>

// ANY component inside can now read it - no props threaded
function Header() {
  const { theme, toggle } = useTheme()
  return <button onClick={toggle}>Theme: {theme}</button>
}
✅
The custom-hook wrapper (useTheme) is a best practice: components import one clean hook instead of both useContext and the context object, and you can add an error if it's used outside the provider.

The critical limitation: re-renders

🚨
Context's big gotcha: when the provider's value changes, every component that consumes that context re-renders — even if it only uses a part that didn't change. There's no built-in way to subscribe to just a slice. For values that change often and are consumed widely, this causes performance problems. This single limitation is the main reason dedicated store libraries (Zustand, Redux) exist.

What Context is and isn't for

Great for ContextReach for a store instead
Theme (light/dark)Fast-changing state read everywhere
Current user / authLarge, complex global state
Language / localeNeed devtools / time-travel
Anything that rarely changesNeed fine-grained subscriptions
💡
Common misconception: "Context is a state manager." Not quite — Context is a delivery mechanism (it passes a value down without props). The actual state still lives in a useState/useReducer somewhere. Zustand/Redux are true state managers and handle delivery efficiently. You can mitigate Context re-renders by splitting into multiple small contexts (e.g. separate ThemeContext and AuthContext), so a change in one doesn't re-render consumers of the other.

Redux family · 04

🏪Redux core concepts

Redux is the most famous state library. It puts all your global state in one central store, changed only through a strict, predictable flow. We'll learn classic Redux first to understand the model — then Section 05 shows the modern way (Redux Toolkit) you'll actually write.

The one-way data flow

The Redux loop
  Component dispatches an ACTION
        |  { type: 'cart/itemAdded', payload: item }
        v
  REDUCER receives (state, action) -> returns NEW state
        |
        v
  STORE holds the single source of truth
        |
        v
  Components SUBSCRIBED to that state re-render
        |
        +-----------< user interacts again <-----------+

  Data flows ONE direction. Every change is traceable.

The vocabulary

TermWhat it is
StoreThe single object holding all global state
ActionA plain object describing what happened: { type, payload }
ReducerPure fn (state, action) => newState — same as useReducer
DispatchThe function to send an action to the store
SelectorA function that reads a slice of state: s => s.cart.total
A bank

You can't reach into the vault and grab cash (no direct state mutation). You fill out a slip (an action: "withdraw $50"), hand it to the teller (dispatch), who follows strict rules (the reducer) to update the ledger (the store). Every transaction is recorded and auditable — that's Redux's predictability and its famous time-travel debugging.

The three principles

  • Single source of truth — all state in one store, so there's one place to look.
  • State is read-only — you never mutate it directly; you dispatch an action.
  • Changes via pure reducers — predictable, testable, no surprises.
⚠️
Classic Redux's reputation for boilerplate is real and deserved. Plain Redux makes you write action type constants, action creator functions, immutable update logic by hand, and lots of wiring — a simple counter could span several files. This pain is exactly why Redux Toolkit (next section) was created and is now the official, recommended way to use Redux. You should essentially never write classic Redux by hand today — but understanding this flow makes Toolkit obvious.
🧠
What Redux is genuinely great at: large apps with complex, interconnected global state; powerful DevTools (inspect every action, time-travel through state changes); a huge ecosystem; and predictability that scales to big teams. What it's overkill for: small apps, or state that's mostly server data (use a query library) or mostly local (use useState).

Redux family · 05

🛠️Redux Toolkit (RTK)

Redux Toolkit is the official, modern way to write Redux. It keeps the exact same concepts from Section 04 but removes nearly all the boilerplate. If you use Redux today, you use RTK — full stop.

What RTK fixes

Classic Redux painRTK solution
Action type constants by handAuto-generated from your reducers
Action creator functions by handAuto-generated
Manual immutable updates (spread everywhere)Write "mutating" code; Immer makes it immutable
Store setup wiringOne configureStore call (DevTools included)
Many files per featureOne "slice" file

The heart of RTK: createSlice

A slice bundles a piece of state with its reducers and auto-generates the actions. One file replaces what used to be several.

features/cart/cartSlice.js
import { createSlice } from '@reduxjs/toolkit'

const cartSlice = createSlice({
  name: 'cart',
  initialState: { items: [], total: 0 },
  reducers: {
    // looks like mutation, but Immer makes it safely immutable!
    itemAdded(state, action) {
      state.items.push(action.payload)
      state.total += action.payload.price
    },
    itemRemoved(state, action) {
      state.items = state.items.filter(i => i.id !== action.payload)
    },
    cartCleared(state) {
      state.items = []; state.total = 0
    }
  }
})

// actions are auto-generated from the reducer names!
export const { itemAdded, itemRemoved, cartCleared } = cartSlice.actions
export default cartSlice.reducer
🧠
The "mutating" magic: inside createSlice, RTK uses a library called Immer. You write state.items.push(...) as if mutating, but Immer tracks your changes and produces a brand-new immutable state behind the scenes. You get simple code AND Redux's immutability rules. This eliminates the #1 source of classic-Redux bugs and boilerplate.

Setting up the store

app/store.js
import { configureStore } from '@reduxjs/toolkit'
import cartReducer from '../features/cart/cartSlice'

export const store = configureStore({
  reducer: {
    cart: cartReducer,   // state.cart is owned by the cart slice
    // user: userReducer, products: productsReducer, ...
  }
})
// DevTools, thunk middleware, good defaults - all configured automatically
main.jsx - provide the store
import { Provider } from 'react-redux'
import { store } from './app/store'

<Provider store={store}>
  <App />
</Provider>

Using it in components

Cart.jsx
import { useSelector, useDispatch } from 'react-redux'
import { itemAdded, itemRemoved } from './cartSlice'

function Cart() {
  // useSelector READS a slice of state (re-renders only when it changes)
  const items = useSelector(state => state.cart.items)
  const total = useSelector(state => state.cart.total)

  // useDispatch gives you the dispatch function
  const dispatch = useDispatch()

  return (
    <div>
      Total: ${total}
      <button onClick={() => dispatch(itemAdded({ id: 1, price: 50 }))}>
        Add item
      </button>
    </div>
  )
}
✅
Two hooks are all you need in components: useSelector to read state (it re-renders the component only when the selected slice changes — solving Context's re-render problem), and useDispatch to send actions. Notice the whole feature is now one slice file plus these two hooks — a fraction of classic Redux.

Redux family · 06

⌛Async with Thunks

Reducers must be pure — they can't make API calls. So how does Redux handle async work like fetching data? With middleware. The default, built into RTK, is the thunk.

What a thunk is

A thunk is simply a function you can dispatch (instead of a plain action object) that's allowed to do async work and dispatch real actions when it's done. The name just means "a function that wraps some work to do later." RTK's createAsyncThunk handles the common pattern for you.

createAsyncThunk - fetch users
import { createAsyncThunk, createSlice } from '@reduxjs/toolkit'

// defines an async action. RTK auto-creates pending/fulfilled/rejected
export const fetchUsers = createAsyncThunk(
  'users/fetch',
  async () => {
    const res = await fetch('/api/users')
    return res.json()   // becomes action.payload on success
  }
)

const usersSlice = createSlice({
  name: 'users',
  initialState: { list: [], status: 'idle', error: null },
  reducers: {},
  // handle the 3 lifecycle actions the thunk dispatches
  extraReducers: (builder) => {
    builder
      .addCase(fetchUsers.pending,   (s) => { s.status = 'loading' })
      .addCase(fetchUsers.fulfilled, (s, a) => { s.status = 'done'; s.list = a.payload })
      .addCase(fetchUsers.rejected,  (s, a) => { s.status = 'error'; s.error = a.error.message })
  }
})
Dispatching it from a component
const dispatch = useDispatch()
const { list, status } = useSelector(s => s.users)

useEffect(() => { dispatch(fetchUsers()) }, [])

if (status === 'loading') return <Spinner />
💡
Why three lifecycle actions? Every async request has three moments: it started (pending → show a spinner), it succeeded (fulfilled → store the data), or it failed (rejected → show an error). createAsyncThunk generates all three for you, so you just decide what each does to the state.
🚀
Thunks vs Sagas (preview): thunks are simple and cover the vast majority of async needs — fetch, then dispatch. They're the right default. Redux-Saga (next section) is a heavier alternative for complex async orchestration: cancelling in-flight requests, debouncing, retrying, reacting to other actions, long-running background flows. Most apps should use thunks; reach for Saga only when the async logic is genuinely complex.

Redux family · 07

🎥Redux-Saga

Redux-Saga is a middleware for handling complex async flows in Redux. Where thunks say "do this async thing then dispatch," Saga lets you describe elaborate, long-running, cancellable processes as if they were step-by-step scripts. It's powerful but heavier — learn when (and whether) you need it.

The core idea: generator functions

Saga is built on JavaScript generator functions (function* with yield). A generator can pause at each yield and resume later — perfect for describing async steps. Saga runs these generators and handles the actual async work, so your saga code stays pure and testable.

A saga that watches for an action and fetches data
import { call, put, takeLatest } from 'redux-saga/effects'

// the worker saga: does the actual fetching
function* fetchUsersSaga() {
  try {
    // 'call' = run this function and wait for it
    const res = yield call(fetch, '/api/users')
    const data = yield call([res, 'json'])
    // 'put' = dispatch an action to the store
    yield put({ type: 'users/loaded', payload: data })
  } catch (err) {
    yield put({ type: 'users/failed', error: err.message })
  }
}

// the watcher saga: listens for the trigger action
function* rootSaga() {
  // takeLatest: on each 'users/fetch', run the worker -
  // and CANCEL any previous run still in progress (no race conditions!)
  yield takeLatest('users/fetch', fetchUsersSaga)
}

The "effects" vocabulary

EffectMeans
call(fn, args)Call a function (usually async) and wait for the result
put(action)Dispatch an action to the store
takeLatest(type, saga)Run the saga on each matching action; cancel prior runs
takeEvery(type, saga)Run the saga on every matching action (no cancelling)
select(selector)Read from the current store state
delay(ms)Wait (for debouncing, polling, retries)
cancel / raceCancel tasks / race two effects against each other
A film director's shooting script

A thunk is a stagehand: "go fetch this prop, then come back." A saga is a full director's script that can say "start this scene, but if the lead actor leaves, cancel and reshoot; wait 3 seconds between takes; if it fails twice, try a stunt double." Saga shines when the choreography of async events is complex — cancellation, debouncing, retries, reacting to other events.

What Saga is genuinely good at

  • Cancellation — takeLatest auto-cancels stale requests (e.g. fast typing in search).
  • Debounce / throttle — built-in via delay and effect patterns.
  • Complex sequences — "do A, wait for B, then if C do D" orchestration.
  • Reacting to actions — one action can trigger a whole background flow.
  • Testability — because sagas yield plain effect descriptions, you can test the flow without running real async.
⚠️
The honest verdict on Saga today: it's powerful but has a steep learning curve (generators, the effects API) and adds significant complexity. For ordinary "fetch and store" needs, thunks are simpler and recommended — and for server data specifically, RTK Query (next) eliminates most async code entirely. Reach for Saga only when you have genuinely complex async orchestration that thunks struggle with. Many modern teams never need it. Knowing what it's for matters more than using it everywhere.

Redux family · 08

📥RTK Query

RTK Query is part of Redux Toolkit and handles server state — fetching, caching, and syncing API data — with almost no code. If you're already in the Redux ecosystem, it's the answer to "how do I fetch data" and it makes most thunks/sagas unnecessary.

The problem it solves

Recall the key insight: server data isn't really "state" you own — it's a cached copy of remote data. Managing it with manual thunks means writing loading flags, caching, refetching, and dedup by hand. RTK Query (like TanStack Query from your HTTP course) does all of that automatically. You just describe your endpoints.

Define an API slice
import { createApi, fetchBaseQuery } from '@reduxjs/toolkit/query/react'

export const api = createApi({
  reducerPath: 'api',
  baseQuery: fetchBaseQuery({ baseUrl: '/api' }),
  endpoints: (builder) => ({
    // a query (read)
    getProducts: builder.query({ query: () => '/products' }),
    // a mutation (write)
    addProduct: builder.mutation({
      query: (body) => ({ url: '/products', method: 'POST', body })
    })
  })
})

// RTK Query AUTO-GENERATES hooks from your endpoints!
export const { useGetProductsQuery, useAddProductMutation } = api
Use the auto-generated hooks
function Products() {
  // fires the request, caches it, gives you all states - one line
  const { data, isLoading, error } = useGetProductsQuery()
  const [addProduct] = useAddProductMutation()

  if (isLoading) return <Spinner />
  return data.map(p => <div key={p.id}>{p.name}</div>)
}
✅
This looks just like TanStack Query from your HTTP course — and that's the point. Both solve server state the same way: describe endpoints, get auto-generated hooks with caching, loading/error states, dedup, and background refetching. The difference: RTK Query lives inside Redux (great if you already use Redux), while TanStack Query is standalone (great if you don't).

RTK Query vs TanStack Query

RTK QueryTanStack Query
Part ofRedux ToolkitStandalone
Use whenAlready using ReduxNot using Redux
Caching, dedup, refetchYesYes
Works with any frameworkReact-focusedReact, Vue, Svelte, Solid
🧠
The big-picture takeaway: once you use a query library (RTK Query or TanStack Query) for server data, your remaining "global client state" is usually tiny — theme, auth, a few UI flags. That small remainder rarely justifies full Redux, which is a major reason lightweight tools like Zustand (next) have become so popular.

Modern minimalist · 09

🐫Zustand

Zustand (German for "state") is a tiny, modern state library that's become hugely popular as a simpler alternative to Redux. It gives you a global store with almost no boilerplate — no providers, no actions, no reducers required.

The whole thing is one function

You create a store with create, defining both the state and the functions that change it in one place. Components subscribe by calling the hook with a selector.

store.js - the entire store
import { create } from 'zustand'

const useCartStore = create((set) => ({
  // state
  items: [],
  total: 0,

  // actions live right next to the state - set() updates it
  addItem: (item) => set((state) => ({
    items: [...state.items, item],
    total: state.total + item.price
  })),
  clear: () => set({ items: [], total: 0 })
}))

export default useCartStore
Using it - no Provider needed!
function Cart() {
  // select ONLY what you need - re-renders only when THAT changes
  const total = useCartStore(s => s.total)
  const addItem = useCartStore(s => s.addItem)

  return <button onClick={() => addItem({ id: 1, price: 50 })}>
    Total: ${total}
  </button>
}

// any other component reads the SAME store - automatically in sync
function HeaderCartCount() {
  const count = useCartStore(s => s.items.length)
  return <span>{count}</span>
}

Why people love it

Strengths
  • Almost no boilerplate
  • No Provider to wrap the app
  • Selector-based: fine-grained re-renders by default
  • Tiny (~1KB)
  • Works outside React too
  • Easy async (just await inside an action)
Trade-offs
  • Less rigid structure than Redux
  • Smaller (though solid) ecosystem
  • DevTools good but less deep than Redux's
  • Freedom can mean inconsistency on big teams
✅
Async is trivial — no thunks or sagas needed. An action can just be async: fetchUser: async (id) => { const u = await api.get(id); set({ user: u }) }. This simplicity, plus selector-based subscriptions that avoid Context's re-render problem, is why Zustand is the default global-state pick for many modern apps.
🧠
Redux vs Zustand in one line: Redux is structured, ceremonious, and battle-tested for large complex state with deep tooling; Zustand is minimal, flexible, and delightful for most apps' modest global state. With server data in a query library, most apps' remaining global state is small — which is exactly Zustand's sweet spot.

⚛️Jotai & the atomic model

Jotai takes a different philosophy: instead of one big store, you build state from many tiny independent pieces called atoms. It's "bottom-up" state — the opposite of Redux's one-big-store, "top-down" approach.

What an atom is

An atom is the smallest unit of state — a single value any component can read and write. You compose many atoms instead of one monolithic store. A component subscribes to exactly the atoms it uses, so re-renders are naturally minimal.

Jotai - atoms
import { atom, useAtom } from 'jotai'

// define atoms - each a single piece of state
const countAtom = atom(0)
const nameAtom = atom('Alice')

// derived atom - computed from others, auto-updates
const doubledAtom = atom((get) => get(countAtom) * 2)

function Counter() {
  // useAtom works like useState, but the state is shared globally
  const [count, setCount] = useAtom(countAtom)
  const [doubled] = useAtom(doubledAtom)

  return <button onClick={() => setCount(c => c + 1)}>
    {count} (doubled: {doubled})
  </button>
}
Lego bricks vs one big sculpture

Redux is one large pre-formed sculpture — all state in a single shaped store. Jotai is a box of Lego bricks (atoms) — you grab just the pieces you need and snap them together; derived atoms are bricks made from other bricks. Components hold only the bricks they use, so changing one brick doesn't disturb the rest.

💡
Atomic feels like useState that's shared. useAtom has the same [value, setValue] shape as useState — the difference is the value lives outside the component and any other component using the same atom shares it. This makes Jotai very easy to adopt incrementally. Recoil (from Meta) pioneered this atomic model; Jotai is the lighter, more popular successor today.

🧩Others: Recoil, MobX, Valtio, XState

A few more libraries you'll encounter. You don't need to learn all of them — but knowing what each is for helps you read codebases and make informed choices.

Recoil

Meta's atomic state library — the original "atoms and selectors" approach that inspired Jotai. Powerful, but development has slowed and the community has largely shifted to Jotai for new projects. If you see Recoil, read it as "atomic state, like Jotai."

MobX

A different paradigm: observable state with automatic tracking. You mark state as observable, and MobX automatically re-renders any component that reads it — no selectors, no actions, no manual subscriptions. It feels almost magical: just mutate the state and the UI updates.

MobX flavor
import { makeAutoObservable } from 'mobx'

class CartStore {
  items = []
  constructor() { makeAutoObservable(this) }
  add(item) { this.items.push(item) }  // just mutate - MobX tracks it
  get count() { return this.items.length }  // auto-derived
}

MobX trades Redux's explicit predictability for convenience and less code. Great for OOP-style codebases; the "magic" can make data flow harder to trace than Redux's explicit actions.

Valtio

From the same author as Zustand — a proxy-based store where you mutate a plain object and components reading it update automatically (MobX-like simplicity, lighter weight). const state = proxy({ count: 0 }) then just state.count++.

XState

Not a typical store — a state machine library. You define explicit states ("idle", "loading", "success", "error") and the allowed transitions between them. Excellent for complex UI flows where you want to make impossible states impossible (e.g. can't be "loading" and "error" at once). Heavier, but unmatched for intricate workflows like multi-step forms, wizards, and media players.

LibraryParadigmOne-line take
RecoilAtomicMeta's atoms; mostly superseded by Jotai
MobXObservable / reactiveMutate freely, auto-tracked; less explicit
ValtioProxy / mutableZustand author's "just mutate" store
XStateState machineExplicit states + transitions for complex flows
🧠
You don't need to learn all of these. The landscape clusters into a few ideas: one big store (Redux), simple store (Zustand), atoms (Jotai/Recoil), observables (MobX/Valtio), and state machines (XState). Understanding the paradigm lets you pick up any specific library quickly. For most apps, the choice is really just Context + Zustand (or Redux Toolkit) + a query library — the rest are situational.

A typical 2025 React app
Server data        TanStack Query (or RTK Query)   <- most of your "state"

The decision flow

How to choose
Is it from the server (API)?
   YES -> TanStack Query (or RTK Query if already on Redux). DONE.
   NO  -> continue...

Does only one component (or a couple nearby) use it?
   YES -> useState / useReducer. DONE.
   NO  -> it's shared client state...

Does it rarely change (theme, locale, auth)?
   YES -> Context API is enough.
   NO  -> you need a real store...

How big/complex is the global state?
   Small-to-medium     -> Zustand (or Jotai for granular/derived).
   Large & complex,
   need deep DevTools   -> Redux Toolkit.
   Complex async
   orchestration        -> RTK + Saga (rare).

Decision guide cards

Reach for Zustand when…
  • You want a global store with minimal fuss
  • Small-to-medium app
  • You dislike boilerplate
  • The default modern pick
Reach for Redux Toolkit when…
  • Large app, many developers
  • Complex interconnected state
  • You want deep DevTools & conventions
  • An existing Redux codebase
Reach for Context when…
  • Theme / auth / locale
  • Value changes rarely
  • You want zero dependencies
Reach for a query lib when…
  • The data comes from an API
  • You want caching & refetch free
  • (Almost always, for server data)
🎓
The most important realization: the "which state library" debate is mostly answered by first separating server state (use a query library) from client state. Once server data is handled by TanStack/RTK Query, the leftover global client state is usually small — and Context or Zustand handles it easily. Redux Toolkit is for when that client state is genuinely large and complex. Match the tool to the kind of state, and the choice makes itself.

The principles that outlast any library

🎓
(1) Separate the four kinds of state — server, global client, local, URL — and match each to its right tool. (2) Server data belongs in a query library, not a global store. (3) Keep state as local as possible; lift only when truly shared. (4) Prefer selector-based subscriptions (Zustand/Redux) over Context for fast-changing widely-read state. (5) Reducers/actions are a pattern, not a library — you saw the same loop in useReducer, Redux, and RTK. (6) Reach for the simplest tool that solves the problem; add power only when pain appears. Libraries change; these don't.

The non-negotiable habits

  • Don't put server data in Redux/Context — use a query library; it removes most of your "state" code.
  • Don't globalize prematurely — most state is local. A global store for one prop one level deep is over-engineering.
  • Don't fear combining tools — Query + Zustand + Context + useState in one app is normal and healthy.
  • Do learn useReducer well — it's the foundation under Redux, and often enough on its own.
FINALCapstone: design a state architecture
Pick a real product (a social app, a SaaS dashboard, a shop) and write its full state plan.
  • List every piece of state and tag it server / global / local / URL.
  • Choose a query library for all server data and justify it.
  • Pick a client-state tool (Context / Zustand / Redux Toolkit) and justify it.
  • Identify what stays in useState and what (if anything) needs URL state.
  • Decide whether you need Saga (probably not) and explain why.
  • Write one sentence per choice explaining the trade-off you accepted.
